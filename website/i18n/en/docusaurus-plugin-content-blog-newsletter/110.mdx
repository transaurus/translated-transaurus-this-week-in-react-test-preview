## React

[Fresh 1.0](https://deno.com/blog/fresh-is-stable?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter)

Le framework Preact officiellement supporté par Deno est maintenant en v1 et considéré comme prêt pour la production. Fresh me rappelle un mélange entre Remix et Astro : pas de JS par défaut, amélioration progressive, application multi-pages avec architecture islands. Aucune étape de build : vous déployez directement votre application TypeScript/Deno sur le Edge en quelques secondes.

[Comment nous avons construit Hydrogen](https://shopify.engineering/how-we-built-hydrogen?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter)

Hydrogen, le méta-framework React de Shopify pour créer des boutiques e-commerce, vient de passer en v1.0 et fut l'un des premiers à s'appuyer sur les React Server Components. Cet article est un retour intéressant sur la création du framework, détaillant divers choix effectués au fil du temps : React Server Components, Vite, Tailwind, préchargement/waterfall, fragments GraphQL...

[Next.js 12.2](https://nextjs.org/blog/next-12-2?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter)

Dernière version de Next.js. Le thème principal est l'adoption progressive du Edge et des API web standardisées : middleware, mais aussi routes API et SSR. Il y a également des améliorations sur les images et un système de plugins SWC en WebAssembly.

[Mon flux de travail fantastique pour les emails HTML](https://www.joshwcomeau.com/react/wonderful-emails-with-mjml-and-mdx/?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter)

Josh Comeau explique comment il rédige ses emails avec MDX, puis utilise MJML, React et Next.js pour générer le rendu HTML final compatible avec tous les clients de messagerie, envoyé via ConvertKit, et également disponible sur le web.

[Flux de données dans Remix](https://remix.run/blog/remix-data-flow?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter)

Le modèle initial de React peut être résumé par `ui = f(state)`. Le problème est que ce modèle ne prend pas vraiment en compte la synchronisation réseau (c'est-à-dire les appels API). Jim explique comment Remix intègre le réseau au modèle React, réduisant ainsi le besoin de gestion d'état local.

[Vous n'avez peut-être pas besoin d'un Effet](https://beta.reactjs.org/learn/you-might-not-need-an-effect?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter)

Nouvelle page de documentation écrite par Dan Abramov, récemment fusionnée sur le site bêta. Les Effets existent pour la synchronisation avec un système externe. De nombreux anti-patterns sont présentés avec des exemples concrets et même des défis. Même les développeurs expérimentés devraient le lire. Certaines choses peuvent surprendre : comme [l'utilisation de setState pendant le rendu](https://twitter.com/sebastienlorber/status/1541793224561463297?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter) 😱. J'ai particulièrement apprécié l'exemple avec `useSyncExternalStore`.

[Quelle est la méthode recommandée pour charger des données dans React 18 ?](https://www.reddit.com/r/reactjs/comments/vi6q6f/what_is_the_recommended_way_to_load_data_for/?utm_campaign=This%20Week%20In%20React&utm_medium=email&utm_source=Revue%20newsletter)

Dan Abramov explique sur Reddit pourquoi la récupération de données via useEffect n'est pas idéale. Cependant, il n'est pas nécessaire de réécrire immédiatement votre application si elle ne présente aucun problème d'expérience utilisateur : ces problèmes ne sont pas nouveaux dans React 18, simplement mieux documentés. Voir aussi les commentaires sur la récupération de données dans React-Native.
