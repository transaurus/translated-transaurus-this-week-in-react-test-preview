**[Postmortem sur le contournement du middleware Next.js](https://vercel.com/blog/postmortem-on-next-js-middleware-bypass)**

A few days ago, Vercel announced a critical 9.1 vulnerability in Next.js middleware system. With a simple but malicious `x-middleware-subrequest` header, you can bypass the middleware, possibly exposing sensitive information. Patches are available for Next.js 12/13/14/15. Note that this vulnerability only affects self-hosted apps using `output: ‘standalone’` and `next start` are affected. Customers of major serverless cloud providers are either not affected (Vercel, Netlify), or protection can be turned on (Cloudflare).

Il est important de noter que même si le middleware Next.js ne devrait pas avoir cette vulnérabilité, ce n'est [pas l'endroit recommandé pour gérer les sessions utilisateur et protéger les routes](https://nextjs.org/docs/pages/building-your-application/authentication#optimistic-checks-with-middleware-optional), et ne devrait pas être la seule ligne de défense dans votre application Next.js. Le middleware est davantage conçu pour effectuer des vérifications optimistes légères, réécrivant/redirigeant éventuellement sans accéder à aucune base de données.

Cet événement de sécurité a provoqué beaucoup de remous dans l'écosystème, sur lesquels je préfère ne pas trop m'étendre. Vercel aurait pu mieux gérer la situation, et ils prévoient de s'améliorer à l'avenir. Je vous laisse vous faire votre propre opinion basée sur diverses ressources, et il y en a trop, donc voici juste un sous-ensemble :

- 📜 [Next.js and the corrupt middleware: the authorizing artifact](https://zhero-web-sec.github.io/research-and-things/nextjs-and-the-corrupt-middleware) : L'article original des chercheurs en sécurité ayant découvert la vulnérabilité.
- 📜 [How to Think About Security in Next.js](https://nextjs.org/blog/security-nextjs-server-components-actions) : Article plus ancien mais toujours pertinent aujourd'hui.
- 📜 [Authorization in Next.js](https://www.robinwieruch.de/next-authorization/) : Robin explique son approche de l'autorisation dans les projets Next.js, au plus près des données sensibles.
- 📜 [You should know this before choosing Next.js](https://eduardoboucas.com/posts/2025-03-25-you-should-know-this-before-choosing-nextjs/) : Le point de vue de l'ingénieur principal de Netlify.
- 🎥 [Theo - Next.js security exploit](https://www.youtube.com/live/UbcrphhSAIs?si=QgihLKnhpyE3RxPO&t=8243) : Un stream d'1h sur ce sujet, expliquant également les erreurs de Vercel.
