## ⚛️ React {#react}

[![Lit React v1.0](/emails/issues/162/lit.jpg)](https://www.npmjs.com/package/@lit/react)

[**Lit React v1.0**](https://www.npmjs.com/package/@lit/react)

Le framework Web Components de Google [Lit vient d'être publié en v3.0](https://lit.dev/blog/2023-10-10-lit-3.0/). En plus d'améliorations du framework (décorateurs, compilateur de templates, Preact Signals), on trouve notamment le package d'intégration `@lit/react` qui a quitté le Lab, est désormais stable et en v1.0.

Cette intégration facilite l'utilisation des Web Components dans une app React (ou Preact) en créant un wrapper qui mappe les props React aux propriétés du custom element, gère les gestionnaires d'événements et facilite le typage TypeScript. Elle comble certaines lacunes de React 18 qui n'intègre pas facilement les Web Components : les props sont mappées vers des attributs plutôt que des propriétés. Voir la [documentation pour le framework React](https://lit.dev/docs/frameworks/react/).

Mais un développeur React moderne peut-il utiliser tout cela facilement ? Lit propose des packages expérimentaux [@lit-labs/ssr](https://www.npmjs.com/package/@lit-labs/ssr), [@lit-labs/ssr-react](https://www.npmjs.com/package/@lit-labs/ssr-react) et [@lit-labs/nextjs](https://www.npmjs.com/package/@lit-labs/nextjs) mais le support est limité et ne fonctionnera pas avec Next.js App Router. Le [Shadow DOM déclaratif](https://developer.chrome.com/articles/declarative-shadow-dom/), qui permet le SSR, n'est pas encore implémenté dans Firefox, mais un polyfill existe. Bref, Lit devrait bien fonctionner sur une app CRA, mais semble encore un peu risqué si vous avez besoin de SSR et SEO.

## 📱 React-Native {#react-native}

[![Utilisation de RCTView et RCTText dans React Native pour des gains de performance](/emails/issues/162/rnperf.jpg)](https://blog.theodo.com/2023/10/native-views-rn-performance/)

[**Utilisation de RCTView et RCTText dans React Native pour des gains de performance**](https://blog.theodo.com/2023/10/native-views-rn-performance/)

Actuellement, les développeurs React-Native discutent beaucoup [sur Twitter](https://twitter.com/FernandoTheRojo/status/1707762822015267219) des performances des composants `Text` et `View`, qui sont des wrappers JS ajoutant une surcharge par rapport aux éléments natifs `RCTText` et `RCTView`. William Candillon avait déjà 🎥 [expliqué ce problème](https://www.youtube.com/watch?v=1D78Tc46Xqo&t=1156s) en 2021, évoquant l'optimisation possible du rendu texte. Cette optimisation est réalisable tant que certaines fonctionnalités comme les écouteurs `onPress` ne sont pas nécessaires. L'article présente un benchmark mesurant l'impact des composants natifs versus réguliers, rapportant un gain d'environ 15%, avec des variations selon la plateforme et l'architecture.

Cette technique devenant mainstream, les développeurs React-Native cherchent à en profiter. On note la création de [react-native-fast-text](https://github.com/peterpme/react-native-fast-text) ou le POC d'un plugin Babel [react-native-optimizer](https://github.com/marklawlor/react-native-optimizer) appliquant automatiquement cette optimisation.
