## ⚛️ React {#react}

**[RFC React : Support de première classe pour les promesses et async/await](https://github.com/reactjs/rfcs/pull/229)**

L'équipe React a décidé de revenir à un modèle `async/await` pour simplifier l'utilisation des Server Components. Problème : il est difficile de prendre en charge `async/await` dans les Client Components. Ils proposent donc d'introduire une API `use(promise)` côté client. Contrairement à d'autres hooks, `use` peut être appelé conditionnellement. Plus tard, il pourrait être utilisé avec d'autres "utilisables" comme le contexte React (en réalité, vous pouvez déjà [utiliser `useContext` conditionnellement !](https://twitter.com/acdlite/status/1581401077915975680)). Utiliser différentes API peut aussi être vu comme une fonctionnalité : cela permet de distinguer les Server Components des Client Components.

Ce RFC suscite des débats intéressants. On s'interroge sur l'intégration avec les bibliothèques de récupération de données. L'utilisation de générateurs semble adaptée à ce cas, mais ils sont difficiles à utiliser en pratique. Nous attendons un nouveau RFC qui introduira une API de cache.

**[La prochaine transition du Web](https://www.epicweb.dev/the-webs-next-transition)**

Kent C. Dodds propose une belle rétrospective des 3 principales architectures web, très bien illustrée et documentée. Il met en avant la dernière architecture, la tendance du moment : Progressively Enhanced Single-Page-Application (Application à Page Unique à Amélioration Progressive). Remix en est une implémentation, avec des abstractions bien pensées qui encouragent à émuler le comportement natif du navigateur. Je ne suis pas tout à fait d'accord qu'un serveur soit absolument nécessaire. [Docusaurus](https://github.com/facebook/docusaurus) est également une implémentation PESPA qui fonctionne bien sans JavaScript.

**[Un terrain de jeu de code de classe mondiale avec Sandpack](https://www.joshwcomeau.com/react/next-level-playground/)**

Josh Comeau vante [Sandpack](https://github.com/codesandbox/sandpack), la boîte à outils CodeSandbox qu'il utilise pour créer des terrains de jeu de code interactifs. Cela semble facile à utiliser et très flexible !

- 📜 [Streaming : cela en vaut-il la peine ?](https://www.builder.io/blog/streaming-is-it-worth-it) : les auteurs de Qwik, Solid et Marko expliquent les avantages du streaming, et pourquoi il est difficile d'adopter cette technique. Dans un contexte React, les bibliothèques utilisées (Helmet, CSS-in-JS, Redux...) doivent être compatibles pour que le streaming soit efficace.
- 📜 [Storybook - Support de première classe pour Vite](https://storybook.js.org/blog/first-class-vite-support-in-storybook/) : 7.0 sera bientôt en version bêta : support de Vite sans configuration, pré-bundling, plus besoin de Webpack, support de nouvelles technologies comme Vue 2, Lit et Svelte...
- 📜 [Passer de React à htmx](https://htmx.org/essays/a-real-world-react-to-htmx-port/) : retour d'expérience (article + vidéo) sur une migration réussie de React vers [htmx](https://htmx.org/). Nous sommes dans un contexte backend Python, leurs conclusions seraient probablement différentes dans un contexte JavaScript où tous les développeurs sont full-stack. Le développeur JS a fait la POC et a perdu son emploi par la suite 😅.

## 🔀 Autre {#other}

**[StackBlitz Codeflow](https://stackblitz.com/codeflow)**

La technologie [WebContainers](https://blog.stackblitz.com/posts/introducing-webcontainers/) de StackBlitz permet d'exécuter Node.js dans le navigateur via WebAssembly. Vous pouvez exécuter VS Code, Next.js ou Docusaurus directement dans votre navigateur, sans passer par un conteneur Docker distant. Codeflow ajoute une couche complète d'intégration avec GitHub qui facilite l'ouverture, la révision ou la modification d'une pull-request en l'exécutant localement de manière sécurisée : extrêmement utile pour les projets open-source.

Ils ont également lancé Web Publisher, une vue d'édition simplifiée pour les fichiers Markdown (éditeur, aperçu, bouton d'enregistrement). Très pratique pour contribuer à la documentation sur Git, même pour les utilisateurs moins techniques. Il y a un véritable aperçu du contenu, contrairement à l'expérience dégradée de l'aperçu Markdown de GitHub. C'est un [besoin important pour les utilisateurs de Docusaurus](https://twitter.com/ericsimons40/status/1580371754974781440).

**[Fontaine - Rechange de police automatique basé sur les métriques de police](https://github.com/unjs/fontaine)**

Permet d'éviter les décalages de mise en page lors du chargement d'une police personnalisée. Rend la police de rechange de la même taille que la police finale, pour une transition fluide qui n'affecte pas la taille des conteneurs de texte. Excellente idée !
