## 💸 Sponsor {#sponsor1}

[![Nlux ― Une nouvelle bibliothèque React/JS pour créer des applications web d'IA conversationnelle](/emails/issues/169/nlux.png)](https://nlux.ai/?c-twr1-h)

**[Nlux ― Une nouvelle bibliothèque React/JS pour créer des applications web d'IA conversationnelle](https://nlux.ai/?c-twr1-h)**

[NLUX](https://nlux.ai?c-twr1-p1) _(pour Natural Language User Experience)_ est une nouvelle bibliothèque [open-source](https://github.com/nluxai/nlux) React/Javascript qui simplifie l'intégration de modèles de langage puissants (LLM) comme ChatGPT dans votre application ou site web. Avec quelques lignes de code, vous pouvez ajouter des capacités d'IA conversationnelle et interagir avec votre LLM préféré.

Fonctionnalités clés 🌟 :

- **Créer des interfaces de chat IA en minutes** - Interfaces d'IA conversationnelle de haute qualité avec peu de code.
- **Composants React & Hooks** - `<AiChat />` pour l'UI et le hook `useAdapter` pour une intégration facile.
- **Adaptateurs LLM flexibles** - `ChatGPT` et `HuggingFace`, avec une API pour **créer votre propre adaptateur** pour tout LLM.
- **Personas de chat personnalisables** - Configurez images, noms et descriptions des bots/utilisateurs pour des interactions personnalisées.
- **Zéro dépendance** - Base de code légère sans dépendances, sauf pour les bibliothèques frontales LLM.

Plus d'informations, exemples et documentation sur [Nlux.ai](https://nlux.ai?c-twr1-p2)

## ⚛️ React {#react}

[![L'avenir de React.use et React.useMemo - une alternative puissante aux sélecteurs de contexte](/emails/issues/169/useMemo.jpg)](https://interbolt.org/blog/react-use-selector-optimization/)

**[The future of React.use and React.useMemo - a powerful alternative to context selectors](https://interbolt.org/blog/react-use-selector-optimization/)**

Colin explique comment le contexte React peut causer des problèmes de performance, l'absence d'une API de sélecteur de contexte, et comment il a optimisé les performances d'une application grâce à l'implémentation tierce [use-context-selector](https://github.com/dai-shi/use-context-selector) de Daishi Kato.

La partie la plus intéressante est un excellent résumé des [récentes discussions sur Twitter](https://twitter.com/sebastienlorber/status/1741218919732429108). Dan Abramov a expliqué que les sélecteurs de contexte ne se composent pas bien. Andrew Clark a partagé le plan d'étendre `useMemo()` et d'autoriser l'utilisation d'autres hooks à l'intérieur. Cela fonctionnerait avec le hook use avec [tout type Usable](https://github.com/acdlite/rfcs/blob/first-class-promises/text/0000-first-class-support-for-promises.md#other-usable-types) (actuellement contexte et promesses, mais peut-être Observables et stores à l'avenir ?), mais aussi [avec des hooks tiers](https://twitter.com/BenJ_Ayc0/status/1742499230525067408) ! Cela résoudrait le problème de sélection fine de manière performante et composable, et serait compatible avec le futur compilateur React Forget.

[![The Two Reacts](/emails/issues/169/2reacts.jpg)](https://overreacted.io/the-two-reacts/)

**[The Two Reacts](https://overreacted.io/the-two-reacts/)**

Excellent article de Dan Abramov, présentant de bonnes raisons pour exécuter du code localement (retour instantané) et à distance (proche de la source de données). Il propose une nouvelle formule React : `UI = f(data, state)`, où `f` représente toute votre application React, s'exécutant dans 2 environnements distincts.

Cela se termine par une excellente question, demandant implicitement si vous inventeriez les React Server Components pour résoudre ce problème.

> Est-il possible de répartir des composants entre votre ordinateur et le mien tout en préservant ce qui fait la force de React ? Pourrions-nous combiner et imbriquer des composants provenant de deux environnements différents ? Comment cela fonctionnerait-il ?
