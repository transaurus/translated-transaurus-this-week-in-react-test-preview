## ⚛️ React {#react}

[![Documentation React - Ajout de onCaughtError / onUncaughtError, amélioration de onRecoverableError](/emails/issues/181/react-errors.jpg)](https://github.com/reactjs/react.dev/pull/6742)

[**Documentation React - Ajout de onCaughtError / onUncaughtError, amélioration de onRecoverableError**](https://github.com/reactjs/react.dev/pull/6742)

Récemment, l'équipe React Core a repensé la gestion des erreurs dans React 19. Ces améliorations sont disponibles dans les versions canary de React 19 et devraient probablement arriver dans Next.js 14.2+ tôt ou tard. Cela inclut notamment de nouveaux callbacks d'API publique disponibles sur la racine React (`createRoot`, `hydrateRoot`) :

- `onCaughtError` 🆕 signale les erreurs interceptées par les limites d'erreur
- `onUncaughtError` 🆕 signale les erreurs non interceptées (évidemment)
- `onRecoverableError` utilise désormais la fonctionnalité ES Error Cause pour signaler la cause originale

La PR liée est la documentation WIP pour ces nouvelles API, incluant de nombreux exemples interactifs. La motivation principale semble être d'afficher une boîte de dialogue d'erreur à l'utilisateur.

Ce qui m'a le plus enthousiasmé, c'est que c'est la première fois que je vois en action le tout nouveau message d'erreur d'hydratation de React affiché dans une boîte de dialogue, comme sur la capture ci-dessus. Cela va grandement améliorer la DX de React !

Le point idéal entre le low code et le développement traditionnel pour les applications riches en CRUD.

En tant que méta-framework React open source conçu pour les applications d'entreprise, [Refine](https://github.com/refinedev/refine) offre une solution headless idéale pour les panneaux d'administration, tableaux de bord et outils internes.

Bien que les outils low code puissent sembler prometteurs initialement, ils échouent souvent à gérer des scénarios complexes en raison de problèmes d'évolutivité, d'options de personnalisation limitées et du risque de verrouillage fournisseur.

Refine permet aux équipes de développer rapidement des applications prêtes pour la production tout en conservant une grande flexibilité.

Cette section est désormais co-écrite avec [Benedikt](https://twitter.com/bndkt)

[![Nouvelle bibliothèque 3D pour React Native en cours de développement](/emails/issues/181/rn3d.jpg)](https://twitter.com/mrousavy/status/1775840325161853389)

🐦 [**Nouvelle bibliothèque 3D pour React Native en cours de développement**](https://twitter.com/mrousavy/status/1775840325161853389)

L'un des aspects les plus inspirants de la communauté RN est d'observer des contributeurs comme Marc Rousavy, William Candillon et Christian Falch repousser les limites du possible avec cette technologie, dépassant largement le simple rendu de texte et de boutons à l'écran. Après les récentes annonces sur Skia, nous avons vu l'état de l'art du rendu 2D dans RN. Désormais, Marc nous offre un aperçu des avancées excitantes dans le rendu 3D avec RN. Apparemment, Margelo développe une nouvelle bibliothèque pour accéder aux API graphiques natives (Metal/Vulkan) depuis RN. Au-delà de la courte [vidéo de démonstration](https://twitter.com/JPoliachik/status/1775866218743787934), j'ai aussi trouvé ce [thread](https://twitter.com/mrousavy/status/1775870100945461277) particulièrement intéressant, où Marc explique leur approche avec une bibliothèque C++ et pourquoi ils n'ont pas choisi Three.js, pourtant remarquable sur le web (React-Three-Fiber prend en charge RN).
