## ⚛️ React {#react}

[![Next.js 13.2](/emails/issues/136/nextjs13.2.jpg)](https://nextjs.org/blog/next-13-2)

**[Next.js 13.2](https://nextjs.org/blog/next-13-2)**

Une belle version avec quelques nouvelles fonctionnalités dont les développeurs de Vercel nous avaient déjà donné un aperçu. La nouvelle API Metadata est utile pour le SEO, fonctionne bien avec le streaming SSR, les Server Components et le nouveau répertoire app. Les nouveaux Route Handlers semblent avoir une bonne utilisabilité : co-localisés, il suffit d'exporter le verbe HTTP. Particulièrement enthousiasmé par les nouvelles fonctionnalités autour de MDX (compilateur Rust, Server Components) : cela devrait rendre les sites de contenu Next.js beaucoup plus légers et rapides à utiliser. Si je comprends bien, Next.js commence à utiliser une étape de codegen TypeScript pour les liens typés (en plus du plugin TypeScript). La compatibilité de Turbopack avec les loaders Webpack peut faciliter l'adoption du nouveau bundler.

La plus grande surprise pour moi est l'introduction d'une API de cache intégrée directement à `fetch()` : permet un peu la même chose que l'ISR mais avec une granularité plus fine, au niveau du composant, pour chaque requête. Il y a même un système d'étiquetage. Next.js reste portable : le cache peut être hébergé sur votre propre infrastructure, ou utiliser un cache LRU local. Bien sûr, cela est optimisé pour le déploiement sur Vercel. Un article dédié donne plus de détails et explique les limites du modèle ISR : [Vercel Cache API: A progressive cache, integrated with Next.js](https://vercel.com/blog/vercel-cache-api-nextjs-cache). Voir aussi le [fil de Sebastian Markbåge](https://twitter.com/sebmarkbage/status/1628845420121128965).

[![The Evolution of Signals in JavaScript](/emails/issues/136/signals.jpg)](https://dev.to/this-is-learning/the-evolution-of-signals-in-javascript-8ob)

**[L'Évolution des Signaux en JavaScript](https://dev.to/this-is-learning/the-evolution-of-signals-in-javascript-8ob)**

Cet article de Ryan Carniato (SolidJS) retrace l'histoire des signaux dans le frontend, à travers KnockoutJS, MobX et bien d'autres solutions.

Les signaux sont restés le sujet tendance de la semaine. Je vous suggère de lire par vous-mêmes les nombreuses discussions qui ont eu lieu sur Twitter :

- Andrew Clark [clarifie sa position](https://twitter.com/acdlite/status/1628811935088013314) et [n'est pas fermé à l'introduction des signaux dans React](https://twitter.com/acdlite/status/1626590880126889984). Le compilateur React Forget [pourrait compiler vers des signaux](https://twitter.com/acdlite/status/1628930249718202369).
- Miško Hevery [explique](https://twitter.com/mhevery/status/1628556944318550016) comment les signaux évitent les problèmes de performance sur le perçage des props.
- Evan You [préfère](https://twitter.com/youyuxi/status/1629636673519894528) un modèle mutable compréhensible à un modèle immutable qui fuit.
- Devon Govett [nous rappelle](https://twitter.com/devongovett/status/1629540226589663233) pourquoi le "flux de données unidirectionnel" a été adopté et souligne certains problèmes avec les signaux.
- Jordan Walke [mentionne](https://twitter.com/jordwalke/status/1629663133039214593) qu'ils avaient envisagé d'utiliser un modèle réactif lors de la création de React. Rejeté car cela changeait la sémantique du langage.
- Dan Abramov [explique](https://twitter.com/dan_abramov/status/1629872956288512007) que l'utilisateur s'attend à voir la même interface pour les mêmes données. Il [explique](https://twitter.com/dan_abramov/status/1629539600489119744) la valeur d'utiliser des "valeurs brutes" (non encapsulées dans un conteneur).
- [Ma position](https://twitter.com/sebastienlorber/status/1630264713845039104) : je partage largement les mêmes vues que l'équipe React et pense que nous devons continuer à innover sur le modèle actuel, malgré l'attrait des signaux à court terme. J'espère qu'éventuellement JavaScript supportera mieux le modèle immutable, et que le compilateur React Forget n'est qu'une étape transitoire.

[![Reanimated 3.0](/emails/issues/136/reanimated3.jpg)](https://blog.swmansion.com/releasing-reanimated-3-0-17fab4cb2394)

**[Reanimated 3.0](https://blog.swmansion.com/releasing-reanimated-3-0-17fab4cb2394)**

Reanimated 3.0 vient d'être publié en version stable, avec une nouvelle fonctionnalité de transitions d'éléments partagés ([démo](https://twitter.com/swmansion/status/1630590156255240193)). Il est désormais possible d'utiliser des valeurs partagées dans les styles en ligne. La bibliothèque a été réécrite pour améliorer les performances, l'expérience développeur (DX) et préparer l'arrivée de nouvelles fonctionnalités. Elle est compatible avec Fabric et l'API de Reanimated 2 pour faciliter la mise à jour, mais l'API de Reanimated 1 a été supprimée.
