## 📣 De l'équipe React {#react-core-team}

[![React Labs: View Transitions, Activity, et plus](/emails/issues/231/labs.jpg)](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more)

**[React Labs: View Transitions, Activity, et plus](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more)**

Les rumeurs selon lesquelles l'équipe React ne se soucie que du serveur sont grandement exagérées.

Dans notre [article React Labs](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more) aujourd'hui, nous avons annoncé que les [View Transitions](https://react.dev/reference/react/ViewTransition) et l'[Activity](https://react.dev/reference/react/Activity) sont maintenant prêts à être testés dans notre canal expérimental, avec une documentation pour chacun. Ces nouvelles fonctionnalités permettent à React côté client d'animer les transitions d'interface, et de masquer/afficher des éléments d'UI lors de la navigation côté client. Elles fonctionnent avec les fonctionnalités serveur mais visent à résoudre des problèmes côté client.

Nous avons également [partagé des mises à jour](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more#features-in-development) sur les domaines problématiques que nous ciblons, impactant les applications rendues côté client, incluant de meilleurs outils de profilage, la compilation automatique des dépendances d'effets, une extension IDE potentielle, les références de fragments et même un store concurrent. Ces zones problématiques que nous voulions adresser depuis longtemps peuvent enfin être traitées grâce à des fonctionnalités récentes comme le React Compiler et l'API `use`.

Ces fonctionnalités sont encore précoces et pourraient changer significativement avant d'être prêtes. Vous pouvez suivre notre progression, et cette newsletter est un excellent moyen de rester informé. Mais ne stressez pas pour tout apprendre - laissez-nous d'abord cuisiner.

Quand nous serons plus confiants dans le déploiement d'une fonctionnalité, nous l'annoncerons sur le blog et publierons sa documentation, comme nous le faisons aujourd'hui pour les View Transitions et Activity.

Essayez les nouvelles fonctionnalités expérimentales et découvrez les domaines sur lesquels nous travaillons dans le [billet de blog](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more).

\-[Ricky Hanlon](https://bsky.app/profile/ricky.fm)

## 📱 React-Native {#react-native}

[![Legend List 1.0](/emails/issues/231/legend-list.png)](https://bsky.app/profile/jayz.us/post/3lni5ofbuvs2s)

**[Legend List 1.0](https://bsky.app/profile/jayz.us/post/3lni5ofbuvs2s)**

Rendre les longues listes performantes est un problème difficile, et l'ancienne architecture de React Native ne facilitait pas les choses avec son calcul de mise en page asynchrone. Nous nous souvenons que Margelo a abandonné son projet `WishList`.  
Legend List est maintenant en version 1.0 et promet d'être encore plus rapide que `FlashList`, tout en gérant les "cas difficiles" : listes infinies bidirectionnelles, tailles d'éléments variables… Et il est compatible avec l'ancienne architecture (bien que moins performant et avec d'éventuels bugs).

Comme Seb l'a mentionné la semaine dernière, une [v2 de `FlashList`](https://github.com/Shopify/flash-list/tree/new-rlv-prototype) est en cours de développement, qui devrait être plus facile à utiliser (pas besoin de spécifier estimatedItemSize par exemple).

Heureusement, tous reprennent l'API de `FlatList`, donc il est facile de passer de l'un à l'autre 🙂.
