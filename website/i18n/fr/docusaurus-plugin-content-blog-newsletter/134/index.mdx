## ⚛️ React {#react}

[![Netlify Acquires Gatsby Inc. to Accelerate Adoption of Composable Web Architectures](/emails/issues/134/netlify-gatsby.jpg)](https://www.netlify.com/press/netlify-acquires-gatsby-inc-to-accelerate-adoption-of-composable-web-architectures/)

**[Netlify Acquires Gatsby Inc. to Accelerate Adoption of Composable Web Architectures](https://www.netlify.com/press/netlify-acquires-gatsby-inc-to-accelerate-adoption-of-composable-web-architectures/)**

C'est la grande nouvelle de la semaine. Gatsby Inc. racheté par Netlify. Le framework passe désormais sous l'égide de Netlify aux côtés d'Eleventy et SolidJS et continuera d'évoluer. Les fonctionnalités cloud (dont le [Valhalla Content Hub](https://www.gatsbyjs.com/products/valhalla-content-hub/)) seront intégrées aux offres de Netlify. Le nouveau slogan est "architecture composable". Gatsby n'est pas le framework React le plus populaire actuellement, mais je pense que cette acquisition a du sens, notamment pour le hub de contenu et toutes les intégrations CMS sur lesquelles ils ont travaillé. L'offre Gatsby Cloud est plus difficile à adopter qu'une fonctionnalité payante optionnelle sur Netlify, et c'est encore plus vrai pour les sites qui n'utilisent pas Gatsby comme framework.

[![Making state machines global in React](/emails/issues/134/xstate.jpg)](https://stately.ai/blog/making-state-machines-global-in-react)

**[Making state machines global in React](https://stately.ai/blog/making-state-machines-global-in-react)**

Présentation de la nouvelle API `createActorContext` apparaissant dans XState 3.1. Elle permet de faciliter la création de machines à état globales XState, associées à un contexte React. Il est possible d'utiliser un système de sélecteur, ou de fournir une machine à état personnalisée au provider (cela semble pratique pour personnaliser une machine depuis les `props` parentes).

## 📱 React-Native {#react-native}

[![React DOM pour le natif (réduire la fragmentation des API)](/emails/issues/134/rn-web.jpg)](https://github.com/necolas/discussions-and-proposals/blob/reduce-fragmentation/proposals/0000-reduce-fragmentation.md)

**[React DOM pour le natif (réduire la fragmentation des API)](https://github.com/necolas/discussions-and-proposals/blob/reduce-fragmentation/proposals/0000-reduce-fragmentation.md)**

La [RFC](https://github.com/react-native-community/discussions-and-proposals/pull/496) de Nicolas Gallagher (Meta / React-Native-Web) vient d'être mise à jour. L'objectif est d'aligner autant que possible les API de React-Native sur les API Web, y compris les éléments DOM comme `div`, `span` et même `select` ! Le "apprendre une fois, exécuter partout" pourrait progressivement devenir "écrire une fois, exécuter partout". La communauté est assez divisée. Il faudra y réfléchir un peu pour savoir si c'est une bonne idée 😄, et vu l'ampleur de ce projet ambitieux, ce n'est clairement pas pour demain !
