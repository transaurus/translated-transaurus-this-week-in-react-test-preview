## 📣 De l'équipe React {#react-core-team}

[![React Labs: View Transitions, Activity, and more](/emails/issues/231/labs.jpg)](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more)

**[React Labs: Transitions de vue, Activité et plus encore](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more)**

Les rumeurs selon lesquelles l'équipe React ne se soucierait que du serveur sont grandement exagérées.

Dans notre [article React Labs](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more) publié aujourd'hui, nous avons annoncé que les [Transitions de vue](https://react.dev/reference/react/ViewTransition) et l'[Activité](https://react.dev/reference/react/Activity) sont désormais prêtes à être testées dans notre canal expérimental, accompagnées de leur documentation. Ces nouvelles fonctionnalités permettent à React côté client d'animer les transitions d'interface utilisateur, et de masquer/afficher des éléments d'UI lors de la navigation côté client. Elles fonctionnent avec les fonctionnalités serveur mais ciblent spécifiquement les problèmes côté client.

Nous avons également [partagé des mises à jour](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more#features-in-development) sur les domaines problématiques que nous traitons, impactant les applications rendues côté client : meilleurs outils de profilage, compilation automatique des dépendances d'effets, extension IDE potentielle, références de fragments et même un store concurrent. Ces problématiques attendent une solution depuis longtemps, désormais abordables grâce à des fonctionnalités récentes comme le compilateur React et l'API `use`.

Ces fonctionnalités sont encore précoces et pourraient évoluer significativement avant leur disponibilité. Vous pouvez suivre leur développement via cette newsletter. Mais ne stressez pas pour tout apprendre - laissez-nous d'abord finaliser le travail.

Quand nous aurons plus de certitude sur la livraison d'une fonctionnalité, nous l'annoncerons sur le blog avec sa documentation, comme nous le faisons aujourd'hui pour les Transitions de vue et l'Activité.

Essayez les nouvelles fonctionnalités expérimentales et découvrez les domaines sur lesquels nous travaillons dans le [billet de blog](https://react.dev/blog/2025/04/23/react-labs-view-transitions-activity-and-more).

## 📱 React-Native {#react-native}

[![Legend List 1.0](/emails/issues/231/legend-list.png)](https://bsky.app/profile/jayz.us/post/3lni5ofbuvs2s)

**[Legend List 1.0](https://bsky.app/profile/jayz.us/post/3lni5ofbuvs2s)**

Rendre les longues listes performantes est un problème difficile, et l'ancienne architecture de React Native ne facilitait pas les choses avec son calcul de mise en page asynchrone. Nous nous souvenons que Margelo a abandonné son projet `WishList`.
Legend List est désormais en version 1.0 et promet d'être encore plus rapide que `FlashList`, tout en gérant les "cas difficiles" : listes infinies bidirectionnelles, tailles d'éléments variables… Et il est compatible avec l'ancienne architecture (bien que moins performant et avec des problèmes possibles).

Comme Seb l'a mentionné la semaine dernière, une [v2 de `FlashList`](https://github.com/Shopify/flash-list/tree/new-rlv-prototype) est également en cours de développement, qui devrait être plus facile à utiliser (par exemple, sans besoin de spécifier estimatedItemSize).

Heureusement, tous ceux-ci reprennent l'API de `FlatList`, ce qui facilite les changements 🙂.
