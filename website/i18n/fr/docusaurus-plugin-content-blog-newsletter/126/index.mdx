## ⚛️ React {#react}

**[React a-t-il un avenir ?](https://www.infoxicator.com/is-react-going-anywhere)**

Réflexion intéressante de Ruben, après avoir rencontré un CTO e-commerce qui a décidé d'abandonner React pour du Vanilla JS. React est-il vraiment responsable des mauvaises performances de son application ? Adopter une alternative est-il justifié ? React restera-t-il leader dans 5 ou 10 ans ?

**[Amélioration par 4x des Temps d'Interaction React](https://www.causal.app/blog/react-perf)**

L'application tableur cloud de Causal avait des problèmes de performance. Ils nous livrent une analyse détaillée des problèmes et de leur résolution via le profiler Chrome. Certains liés à AG Grid (qu'ils ont patché). D'autres provenant de leur code, principalement liés à la mémorisation. Ils ont aussi essayé les Web Workers, mais la sérialisation était trop coûteuse.

- 📜 [Amélioration progressive du hook useFetcher dans Remix](https://sergiodxa.com/articles/progressively-enhance-the-usefetcher-hook-in-remix): astuce pour prendre en charge les utilisateurs avec/sans JS sur l'API useFetcher de Remix : forcer une redirection pour les utilisateurs sans JS. De même, il est possible de gérer [l'amélioration progressive lors d'un téléchargement de fichier](https://twitter.com/ebey_jacob/status/1596223682246234113). Tout ceci n'est probablement pas essentiel pour de nombreuses applications, mais c'est bon à savoir si nécessaire.
- 📜 [Aperçu de Remix Deferred](https://gist.github.com/jacob-ebey/9bde9546c1aafaa6bc8c242054b1be26): explication du fonctionnement de [defer](https://github.com/remix-run/remix/issues/4486) dans Remix, qui permettra de "téléporter" une promesse du backend vers le frontend. Pour ceux qui souhaitent comprendre les mécanismes internes de Remix.
- 📜 [Comment mettre à jour l'état d'un composant frère dans React ?](https://alexsidorenko.com/blog/react-siblings/): court article illustrant bien le concept de "Remontée d'état" (Lifting state up).
- 📜 [Cours intensif minimal sur TypeScript pour React](https://profy.dev/article/react-typescript): pour démarrer avec TypeScript et React.
- 📜 [Structures de données en JavaScript frontend dans le monde réel - avec des exemples de code React](https://profy.dev/article/javascript-data-structures): pour se familiariser avec les structures de données dans un contexte React.
- 📜 [ArConnect migre son extension Chrome vers le framework Plasmo](https://blog.plasmo.com/p/migrating-chrome-extension-to-plasmo): retour sur l'utilisation de [Plasmo](https://github.com/PlasmoHQ/plasmo): framework React open-source pour créer des extensions web.
- 📜 [Storybook pour les développeurs full-stack](https://storybook.js.org/blog/storybook-for-full-stack-developers/): bon aperçu de tout ce que l'écosystème Storybook peut faire pour vous.
- 📜 [Solid Final Form – Preuve de concept](https://erikras.com/blog/solid-final-form-poc): Erik explique que sa bibliothèque [Final Form](https://final-form.org/react) (populaire il y a quelques années) est découplée de React et montre qu'il est possible de l'utiliser dans un contexte SolidJS.
- 📜 [Ant Design 5.0 - CSS-in-JS au niveau composant](https://ant.design/docs/blog/css-in-js): le système de design React publie une nouvelle version majeure et explique comment ils ont optimisé leur runtime CSS-in-JS.
- 🎥 [Sur les mérites et limites de React et des applications monopages](https://www.youtube.com/watch?v=ChfjZV-aA_Y): débat de 2 heures entre Alex Russell et Theo Browne. Je n'ai pas encore eu le temps de regarder. Alex est connu pour ses critiques de React et des SPAs, et Theo devrait être un bon défenseur de l'écosystème React.
- 🎥 [Développeur React, vous devez connaître @container !](https://www.youtube.com/watch?v=ypN-Uwshc5M): bonne introduction aux requêtes de conteneur, nouvelle fonctionnalité CSS, présentée dans un contexte React en CSS vanilla + Tailwind. Le support Firefox manque encore, mais il existe un polyfill.
- 🎥 [Bases de RTK Query : Points de terminaison, flux de données et TypeScript](https://egghead.io/courses/rtk-query-basics-query-endpoints-data-flow-and-typescript-57ea3c43): nouveau cours gratuit sur Egghead par Lenz, l'un des créateurs de RTK Query.
- 🧑‍💻 [PR Remix - Ajout du support "saut vers la définition"](https://github.com/remix-run/remix/pull/4689): les contributeurs de tRPC appliquent leurs connaissances TypeScript à Remix pour améliorer son DX.
- 📦 [react-three-editor](https://github.com/pmndrs/react-three-editor): un éditeur de scènes React-Three-Fiber est en cours de création.
- 📦 [Gatsby 5.1](https://www.gatsbyjs.com/docs/reference/release-notes/v5.1/)

## 📱 React-Native {#react-native}

**[Sommet des contributeurs principaux de React Native 2022](https://reactnative.dev/blog/2022/11/22/react-native-core-contributor-summit-2022)**

Une grande partie des contributeurs React-Native s'est réunie début septembre lors de la conférence React-Native EU. Ils ont participé à une série d'ateliers et nous font un compte-rendu. Divers sujets ont été abordés, comme la nouvelle architecture, le bundler Metro ou le workflow de publication.

**[Shopify - Notre solution pour mesurer les temps de rendu React Native](https://shopify.engineering/measuring-react-native-rendering-times)**

Présentation du package [react-native-performance](https://github.com/Shopify/react-native-performance) qui permet de générer des rapports de performances [terrain / Real User Monitoring](https://web.dev/lab-and-field-data-differences/) (au format JSON). Il s'intègre facilement avec React-Navigation. Vous pouvez envoyer ces rapports à un tableau de bord d'analyse de votre choix. Cela permet d'identifier un écran spécifique ayant des problèmes de performance en production. Cela rappelle les métriques web existantes : Core Web Vitals, TTI, INP et les tableaux de bord de performance Vercel/Netlify/Gatsby...

- 📜 [Shopify - Ce que nous avons appris en open-sourçant FlashList](https://shopify.engineering/what-we-learned-from-open-sourcing-flashlist): retour sur un lancement open-source réussi chez Shopify. Le marketing et la planification sont des éléments importants du succès et ne doivent pas être négligés. Voir aussi mon lancement de [Docusaurus 2.0](https://www.indiehackers.com/post/1-on-product-hunt-thanks-to-newsletters-and-good-timing-2c7d8809d9).
- 📜 [Comment construire une app React Native 5x plus vite avec Nitro](https://blog.nitro.build/how-to-build-a-react-native-app-5x-faster-with-nitro-d7e512b18d90): Nitro est un outil (payant, en préversion) qui permet de réduire significativement les temps de build CI React-Native lorsque seul le code JS a changé.
- 📜 [React Native en 2022 et au-delà](https://semaphoreci.com/blog/react-native)
- 📜 [Comment react-native est devenu aussi performant que le natif avec la nouvelle architecture](https://medium.com/ocp-digital-factory/how-react-native-became-performant-as-native-with-the-new-architecture-6e678c3178aa)
- 🐦 [setNativeProps est de retour sur Fabric ?](https://twitter.com/o_kwasniewski/status/1596090790350303233)
