## ⚛️ React {#react}

[![React Labs - Ce sur quoi nous avons travaillé – Février 2024](/emails/issues/174/react-labs.jpg)](https://react.dev/blog/2024/02/15/react-labs-what-we-have-been-working-on-february-2024)

**[React Labs - Ce sur quoi nous avons travaillé – Février 2024](https://react.dev/blog/2024/02/15/react-labs-what-we-have-been-working-on-february-2024)**

Cette semaine a apporté la première mise à jour officielle de React Labs depuis près d'un an, avec des annonces passionnantes. Nous en avons déjà couvert plusieurs ici, mais c'est bon d'avoir un résumé officiel !

React Forget s'appelle 🐦 [désormais React Compiler](https://twitter.com/en_JS/status/1758224340880158955) et il semble que le déploiement en production 🐦 [sur Instagram](https://twitter.com/sebastienlorber/status/1734607171030303186) ait été un succès.

La prochaine version de React sera une version majeure. React 19 est proche, et il y a une certaine [conférence](https://conf.react.dev/) qui se prêterait parfaitement à une telle annonce 😏. La version inclura également une prise en charge intégrée de la gestion des métadonnées de page (support intégré pour ce que React-Helmet faisait pour nous, avec un support approprié du streaming), le chargement des ressources (y compris l'intégration de Suspense) et les Actions pour envoyer des données du client vers le serveur. La surface que les méta-frameworks comme Next.js et Remix doivent couvrir avec des implémentations propriétaires diminue.

L'une des fonctionnalités React à venir les plus sous-estimées, l'API Offscreen (nommée Activity), est temporairement dépriorisée pour se concentrer sur React 19. Nous devrons attendre un peu plus pour exploiter pleinement le modèle concurrent introduit par React 18.

Globalement, c'est une période passionnante pour être développeur React, et nous attendons avec impatience la prochaine version majeure !

[![React Strict DOM](/emails/issues/174/strict-dom.jpg)](https://github.com/facebook/react-strict-dom)

**[React Strict DOM](https://github.com/facebook/react-strict-dom)**

Meta n'a pas seulement annoncé de grandes nouveautés concernant React cette semaine. Aujourd'hui, un tout nouveau dépôt est apparu sur GitHub : [react-strict-dom](https://github.com/facebook/react-strict-dom). Le fichier readme le décrit comme une « intégration expérimentale de React DOM et StyleX visant à améliorer et standardiser le développement de composants React stylisés pour le web et le natif. » Il s'appuie sur les idées de la [RFC « React DOM for Native »](https://github.com/react-native-community/discussions-and-proposals/pull/496) et pourrait, à mon avis, révolutionner notre approche du développement d'applications universelles à l'avenir. Actuellement, l'approche la plus courante pour les applications universelles avec React Native est React Native Web, qui émule les composants et API RN sur le web. Bien que RNW soit excellent, il donne souvent l'impression d'adapter une plateforme à une autre. La prémisse de react-strict-dom est d'offrir une bibliothèque DOM unique prenant universellement en charge le web, iOS et Android (et potentiellement d'autres plateformes à l'avenir). L'API des éléments est basée sur le web, ce qui explique la syntaxe très proche du HTML visible dans la capture d'écran ci-dessus.

Comme toujours, Dan Abramov excelle à exprimer l'idée de manière concise dans un 🐦 [tweet](https://twitter.com/dan_abramov2/status/1760245165703184516) :

> « On dirait un successeur spirituel de RNW, mais abordé dans l'autre sens (les API web ayant gagné, RN développe la prise en charge d'un sous-ensemble strict). Concrètement, vous pourrez afficher un `<div>` dans RN. »

Le chemin reste long au vu de l'actuel [état de compatibilité](https://github.com/facebook/react-strict-dom/blob/main/packages/react-strict-dom/COMPATIBILITY.md), mais cela constitue une étape passionnante vers un React plus universel ! Meta l'utilise déjà ! Voir aussi ce 🐦 [thread de Lorenzo sur l'implication de Microsoft](https://twitter.com/Kelset/status/1760367655423549480).
