## ⚛️ React {#react}

[![Bun 1.0](/emails/issues/159/bun.jpg)](https://bun.sh/blog/bun-v1.0)

[**Bun 1.0**](https://bun.sh/blog/bun-v1.0)

本周不可能错过它：Bun 刚刚发布了稳定的 v1.0 版本。虽然这消息不直接关联 React，但迟早会影响所有 React 开发者。

Bun 是基于 Webkit 并用 Zig 编写的 JavaScript/TypeScript 完整工具链。它为我们当前使用的整套前端工具提供了全新超高速替代方案：Node.js、npx、Babel、esbuild、swc、webpack、Jest、Vitest、npm、Yarn、pnpm... 其性能基准令人印象深刻且真实可信。

它与现有工具的兼容性以及对ESM/CJS混合支持的特性，使其能够轻松地作为即插即用的替代方案。Bun通常能够直接运行你的代码而无需任何修改（否则会被[🐦视为一个错误](https://twitter.com/jarredsumner/status/1700497869160268155)）。诸如Next.js、Remix和Astro等React框架已得到支持。将其作为npm/Yarn/pnpm或Jest/Vitest的替代品可能是一个不错的切入点，风险也不大。

## 📱 React-Native {#react-native}

[![Static Hermes](/emails/issues/159/hermes.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

[**Static Hermes**](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

Static Hermes 在 React-Native EU 大会上于上周宣布。我未能亲临现场，因此尝试通过阅读幻灯片来阐释我的理解，因为相关视频尚未上传至 YouTube。

Static Hermes 是一个实验性项目，允许您将部分 TypeScript（或 Flow）代码选择性地编译为原生代码。与基于字节码解释的 Hermes 常规模式相比，这带来了显著的性能提升（10-20 倍）。同时支持直接从 TypeScript 代码无缝集成原生 API（零成本 FFI，比 JSI 封装快 15-80 倍）。

要实现此功能，TypeScript 必须首先成为"健全"（sound）语言。Static Hermes 将修改 JavaScript 的语义，确保声明类型与运行时类型完全匹配。这张幻灯片清晰地阐释了该概念：

[![Static Hermes 使 TypeScript 变得健全](/emails/issues/159/hermes2.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement?slide=14)

这个非常有趣的想法，或许未来能在 React-Native 生态系统之外产生更广泛的影响？
