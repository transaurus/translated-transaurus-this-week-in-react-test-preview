# This Week In React - 策展人职位

我最初创建 [This Week In React](https://thisweekinreact.com/) 是为了通过精选简报帮助 React 开发者及时了解生态系统动态。

长期目标是成为面向 React 和 React Native 开发者的真正社区驱动媒体（类似于 CSS-Tricks）。我希望它不仅仅是一份包含链接列表的周报。

为此，我需要委派工作，并希望招募简报协作者来协助制作每周内容。

这是**真实工作**：您将获得报酬，需要**可靠**，并愿意为单期简报投入 **14-18 小时的工作量**。

## 对 This Week In React 的益处

本次招募的主要目标是：

- 在我休假时提供可靠备份，避免倦怠
- 通过每年 50+ 期内容展现专业媒体形象
- 增强媒体社区驱动属性
- 通过轮换作者提升媒体多样性
- 腾出时间进一步发展媒体
- 确保简报质量持续保持高水平

五年来我几乎独自承担这项工作，初期的新鲜感可能成为长期负担。我希望自主决定何时发布简报，而非被迫执行。希望将自己从等式中移除，使项目最终能超越我个人而延续。

## 客座策展人获益

您将获得的主要收益：

- 每期简报获得**合理报酬**
- 提升**个人品牌**及公司**知名度**（[客座策展人撰写案例](https://thisweekinreact.com/newsletter/206)）
- 回馈社区，贡献于实用项目

如果您有创新想法，我愿探讨更多收益方案。前提是达成双赢。

## 入职流程

截至目前，入职流程包含：

- 您需要自我介绍（或介绍贵公司的员工），并说明您认为自己是合适人选的原因

- 我们共同审阅成果。虽然不可能完美，但只要我对发送的邮件内容不感到尴尬，就说明流程可行且未来可优化
- 我们可以确定您负责的其他日期

## 工作流程

**摘要**：简报筛选流程通常在周一启动，周三傍晚发送（约UTC 17点 / PST 上午9点）。您需确保邮件在周三前100%准备就绪。整个过程约耗时16小时，可由两位开发者协同完成。

**主要步骤：**

- 提前数周规划您将负责的简报期次
- 通过X平台、RSS、网页及自然渠道初步筛选有价值链接
- 将链接添加至协作Google文档
- 审阅链接内容，格式化并过滤低价值条目
- 对重点链接进行专业点评
- 精选头条新闻
- 选择与本周动态相关的趣味模因
- 创建包含React/RN关键词的标题
- 撰写引言文本
- 转换为MDX格式
- 推送至Git并发布在线版本
- 在社交平台协助推广简报

**以下仍由我负责：**

- 管理简报赞助商及广告位投放
- 将网页版简报转换为邮件版本并执行发送

**待定事项：**

- 在X平台发布推文串或文章
- 在Bluesky、Reddit、LinkedIn等平台发布内容

我们会提前数月查看[简报公共日历](https://thisweekinreact.com/calendar)确定您的供稿日期。一旦日期确定，您即成为该期简报的指定作者，并**必须在日程中预留约2个工作日**进行内容制作。

## 编辑方针

**核心原则：**

- 面向**中高级React/React Native开发者**，但不限于React相关内容
- 强调**新鲜内容**（超过1个月的内容视为过时）
- 捕捉**行业弱信号**以预见未来趋势
- 聚焦**事件驱动内容**（如新版本发布、新项目启动、文档更新）
- 成为**生态脉搏监测器**而非React npm包发现工具

**优先选择：**

- 专注**单一概念**并**深度解析**的内容
- 涵盖**React核心技术**且具有广泛适用性的内容
- 呈现**新颖**、**创新**或**被低估**技术方案的内容
- 以出色形式展现**细分领域**及**弱信号**的深度内容
- 具有**卓越呈现效果**的内容（如交互式技术文章）

**应避免内容：**

- 过滤由AI生成的内容（出于SEO考量）、列表式文章及其他低质量内容。
- 过滤需付费订阅、身份验证或阅读体验差的内容。
- 过滤面向初级开发者的基础教程、逐步操作指南、无实质增量的官方文档复述内容。
- 过滤推广付费产品的教学内容（此类需求由赞助商满足），除非其非过度营销且包含有价值的React核心技术教学。
- 过滤大多数UI库和样板文件（数量过于庞大）。例外情况：证明具有高关注度、长期维护承诺或权威来源的项目。不报道此类项目的"首发"，但可能在其主流化并证明社区价值后（如v10版本）进行覆盖。
- 过滤过弱的弱信号：若无法用简单语言解释其影响，或预估仅极少数人受益，则可能过早或不值得收录。
- 尽量避免X/Twitter串/讨论链接（需登录）。优先选用站外链接。若单条推文包含完整信息、链接或视频演示则可保留。

例外情况：具有卓越呈现效果的内容。例如用交互式文章解释基础React概念是可接受的——这是目标受众（资深开发者）可能分享给初级同事的优质资源。

## 内容规范

因React生态活跃，通讯包含大量链接。我们不在链接数量上过度筛选，而是优化信息密度下的阅读体验。内容需保持一致性、易浏览性，并尽可能"内联"关键信息。避免诱导点击却令人失望的链接。即使不点击任何链接，阅读通讯本身也应产生价值。

### 版块结构

通讯包含5个主要版块：

- 导读：新闻摘要
- React：面向全体React开发者及Web开发者的内容
- React Native：面向React Native开发者的内容
- 其他：面向更广泛JS生态的内容（含TS/CSS/TC39/HTML/工具链等）
- 趣味板块：1-3则meme或趣味截图，优先选择与React（内部梗）及当周新闻相关的内容

### 头条选择

常规选择：

- 1条React头条
- 1条React Native头条

每条头条通常配顶部横幅图。优先选用具视觉意义的内容（如代码截图），精美社交卡片亦可接受。

有时某版块无明确头条，有时存在双头条。偶尔会为"其他"版块设置头条。

### 格式规范

链接标准格式（多数情况）：

- \<Emoji\> [链接标题](https://example.com)：可选主观注释

若原标题过长、含点击诱饵或大小写异常，可重构标题以保持整体一致性。

版本发布需省略"v"及补丁号（如3.1而非v3.1.9）。在主链中直接用短横线`-`分隔核心更新关键词。若无实质更新内容（常见于补丁版本）则不予收录。链接内为"客观"内容（更新亮点），链接外为"主观"注释（若有）。示例如下：

- 📦 [Astro 5.2 - 支持 Tailwind 4、尾部斜杠重定向、可禁用 React 流式传输](https://astro.build/blog/astro-520/)：流式传输可能与运行时 CSS-in-JS 库不兼容。
- 📦 [Nextra 4.0 - 现采用 App Router](https://the-guild.dev/blog/nextra-4)
- 📦 [React Scan 0.1.2 - 重新渲染历史记录、组件树](https://github.com/aidenybai/react-scan/releases/tag/v0.1.2)

如果是首次亮相的相对较新项目，可省略版本号，用客观的项目描述替换关键词（灵感来源于 GitHub 仓库说明、落地页或 SEO 标签）：

- 📦 [Polygen - 在应用中静态运行 WebAssembly 模块，无需解释](https://github.com/callstackincubator/polygen)：此 Callstack 预览包将 Wasm 转换为 C 并通过 JSI 调用以提供卓越性能。
